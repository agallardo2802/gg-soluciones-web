Modernización y Migración del Sistema de Actas Digitales

Actualización técnica, organización y migración a infraestructura Linux — base estable para operar y crecer.

1. Objetivo

Modernizar, ordenar y migrar el sistema actual de Actas Digitales del Registro Civil, asegurando mayor estabilidad, mantenimiento ordenado y una infraestructura preparada para operar sobre servidores Linux.

El trabajo contempla la actualización técnica del sistema, la organización de los proyectos existentes y la separación de responsabilidades entre servidor de aplicaciones y servidor de base de datos.

2. Situación actual y riesgos

El sistema actual opera sobre una infraestructura de servidor único, sin redundancia ni plan de continuidad ante fallas. Ante una caída del servidor — ya sea por fallo de hardware, error de sistema operativo o problema de red — el Registro Civil queda sin operación hasta restablecer manualmente el servicio, sin garantía de tiempo de recuperación ni protección contra pérdida de datos recientes.

Una interrupción del sistema obliga al Registro Civil a operar en papel, con el consecuente reingreso manual de datos, riesgo de errores en actas y demoras en la atención al público. El costo operativo de un día sin sistema — en horas de personal, reprocesos y atención al público afectada — supera ampliamente el costo diario de este proyecto.

La presente propuesta resuelve la base técnica necesaria para operar de forma estable. Una vez completada, se recomienda avanzar hacia una arquitectura con alta disponibilidad que elimine este punto único de falla.

3. Alcance del trabajo

Modernización técnica del sistema

  • Migración y actualización de librerías
  • Revisión de versiones de backend, frontend y dependencias
  • Ajustes de compatibilidad para ejecución estable
  • Corrección de configuraciones técnicas heredadas
  • Validación funcional posterior a los cambios

Organización de proyectos

  • Ordenamiento de backend, frontend e integraciones
  • Revisión de estructura de configuración por ambiente
  • Separación clara entre código, configuración y archivos operativos
  • Preparación del sistema para despliegues más controlados

Migración a servidores Linux

  • Preparación del entorno productivo sobre Linux
  • Configuración de servidor para APIs y aplicación web
  • Configuración de servidor independiente para base de datos
  • Ajustes de conexión entre aplicación y base de datos
  • Configuración básica de seguridad, logs y respaldos
  • Puesta en marcha y prueba funcional del sistema migrado

10. Etapa 2 — Dockerización, Desarrollo y Backup (propuesta posterior)

Próximo paso recomendado una vez finalizada esta etapa

Con el sistema estabilizado sobre Linux, la Etapa 2 suma dos VMs adicionales y dockeriza toda la infraestructura, dejando el sistema preparado para crecer o migrar a cualquier entorno.

Infraestructura resultante (4 VMs)

VM 1 — ProducciónAplicación web y APIs (Etapa 1)
VM 2 — ProducciónBase de datos (Etapa 1)
VM 3 — DesarrolloEntorno de desarrollo y testing: APIs + base de datos de pruebas, aislado de producción (nueva — Etapa 2)
VM 4 — BackupStorage y respaldo automatizado de VM 1 y VM 2. Sin carga de aplicación — dedicada exclusivamente a retención y recuperación (nueva — Etapa 2)

Alcance de Etapa 2

  • Dockerización completa — aplicación, APIs y base de datos en contenedores en todas las VMs, facilitando la migración futura a cualquier nube o entorno.
  • VM de desarrollo — entorno aislado con APIs y base de datos de pruebas para validar cambios antes de aplicarlos en producción.
  • VM de backup / storage — política de backup automatizado y verificado de ambas VMs productivas, con retención configurable y procedimiento de recuperación documentado.

Esta etapa se cotizará por separado con el sistema ya estabilizado como base.

11. Resultado esperado

Sistema ordenado, actualizado y migrado a una infraestructura Linux separada por responsabilidades — APIs y aplicación en un servidor, base de datos en otro — dejando una base técnica estable para mantenimiento, crecimiento y la incorporación futura de alta disponibilidad.

¿Por qué es fundamental hacerlo ahora?

  • Estabilidad operativa real — el sistema deja de depender de configuraciones frágiles y pasa a correr sobre una base predecible, con separación clara entre aplicación y datos.
  • Reducción de riesgo técnico — las librerías y dependencias desactualizadas acumulan vulnerabilidades conocidas. Cada mes que pasa sin actualizar es un mes más de exposición.
  • Mantenimiento ordenado — con la estructura de proyectos organizada y los ambientes separados, cualquier cambio futuro tarda horas en lugar de días.
  • Base para crecer — la arquitectura propuesta permite incorporar monitoreo, alta disponibilidad, backups automáticos y eventualmente migración a la nube sin reescribir el sistema.
  • Independencia de plataforma — Linux es el estándar de infraestructura moderna. Elimina dependencias de licencias de sistema operativo y habilita el uso de toda la cadena de herramientas DevOps actual.
  • Menor costo de operación a largo plazo — un sistema ordenado tiene menor costo de soporte, menor tiempo de resolución de incidentes y menor riesgo de retrabajo costoso.

¿Qué puede pasar si no se realiza?

El escenario de riesgo más probable no es gradual — es abrupto. Un servidor único sin plan de recuperación no avisa antes de fallar. Cuando falla, el Registro Civil queda sin sistema y sin datos accesibles hasta que alguien resuelva el problema manualmente, sin documentación, sin entorno de pruebas y con dependencias que nadie recuerda cómo configurar.

  • Falla total sin recuperación ordenada — sin separación de responsabilidades ni backups verificados, una caída de hardware puede implicar pérdida de datos o días de inactividad.
  • Deuda técnica creciente — cada cambio sobre un sistema desorganizado es más costoso, más riesgoso y más lento. La deuda no desaparece sola: se acumula.
  • Vulnerabilidades sin parchear — dependencias obsoletas tienen CVEs públicos conocidos. Mantener el sistema sin actualizar es una decisión de riesgo de seguridad activa.
  • Imposibilidad de escalar — sin una arquitectura base ordenada, agregar funcionalidades, nuevos módulos o integraciones requiere refactorizar todo desde cero.
  • Dependencia de personas clave — un sistema sin documentación ni estructura clara solo lo entiende quien lo armó. Si esa persona no está disponible, el sistema se vuelve inoperable.
  • Costo de emergencia vs. costo de prevención — resolver una crisis en producción sin preparación previa cuesta entre 3 y 10 veces más que hacerlo de forma planificada.
